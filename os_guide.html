Introduction to OS Development & The Boot Sector

Where every operating system begins: the first 512 bytes of execution

1.1 What Is an Operating System?

An operating system is the foundational software layer that sits between computer hardware and user applications. Its primary responsibilities include process management (deciding which program runs and for how long), memory management (allocating and protecting RAM for each process), device management (communicating with disks, keyboards, displays, and other peripherals through device drivers), file system management (organizing persistent data on storage devices), and providing a user interface (whether a command-line shell or a graphical windowing system).

When you write a program in C, Python, or any high-level language, you are not directly talking to the hardware. Every printf(), every file open, every memory allocation ultimately translates into a request to the operating system kernel. The kernel is the privileged core that has direct access to hardware. User programs run in a restricted user mode (Ring 3 on x86), while the kernel runs in supervisor mode (Ring 0). This separation is what prevents a buggy program from crashing the entire machine or corrupting another program's data.

In this guide, we will build a complete 32-bit operating system from absolute scratch. We will start with a single 512-byte sector that the BIOS loads from disk. We will learn x86 assembly language from first principles, not assuming any prior knowledge. We will transition the CPU from 16-bit "real mode" (the state it powers on in) to 32-bit "protected mode," set up memory paging, handle hardware interrupts, write device drivers for keyboard and VGA graphics, implement a file system, build a command shell, create a multitasking scheduler, and finally, construct the foundation for a simple programming language and graphical user interface.

Why Assembly?
While most modern OS kernels (Linux, Windows, macOS) are written primarily in C, the lowest-level components — the bootloader, context switch code, interrupt handlers, and early initialization — must be written in assembly. Assembly gives us precise control over every CPU register, every memory address, and every instruction. There is no hidden runtime, no standard library, no compiler making decisions for us. This total transparency is essential for OS development.

1.2 The Boot Process: From Power-On to Boot Sector

When you press the power button on an x86 PC, a remarkably complex sequence of hardware and software events unfolds. Understanding this sequence is the foundation of OS development.

1.2.1 The CPU Reset State

The CPU's internal reset circuit forces all registers to known "power-on" defaults. The most important of these is the instruction pointer. On the original 8086, the CPU started executing at physical address 0xFFFF0 (CS = 0xF000, IP = 0xFFF0). On modern x86 CPUs, after various micro-architectural initialization, control eventually transfers to the platform firmware — what we call the BIOS (Basic Input/Output System) or, on newer systems, UEFI (Unified Extensible Firmware Interface).

The firmware is stored in non-volatile flash memory on the motherboard. Its job is to initialize the hardware to a known working state and then find and load the operating system. This initialization phase is called the POST (Power-On Self Test).

1.2.2 The POST Sequence

POST is a systematic hardware verification and initialization sequence. While implementations vary between BIOS vendors (Award, AMI, Phoenix), the general flow is:

  1. CPU verification: The BIOS tests internal CPU registers, confirming basic arithmetic and logical operations work correctly.
  2. Chipset initialization: The memory controller, I/O controller, and bus bridges are configured.
  3. RAM detection and test: The BIOS probes memory slots, determines installed RAM capacity, and writes test patterns to verify integrity.
  4. Video initialization: The BIOS scans for a video card, executes its option ROM (which contains its own mini-BIOS), and switches to a text display mode so POST messages can appear.
  5. Keyboard and peripheral init: The PS/2 keyboard controller (8042) is reset, the PIT (Programmable Interval Timer) is configured, and the PIC (Programmable Interrupt Controller) is initialized.
  6. PCI bus enumeration: Every PCI and PCIe slot is scanned. Devices with option ROMs have those ROMs executed.
  7. Boot device selection: The BIOS reads the configured boot priority from CMOS memory and attempts each device in turn.

1.2.3 How the BIOS Loads Your Boot Sector

When the BIOS decides to boot from a device (floppy, hard disk, USB), it performs the following exact steps:

  1. It reads exactly 512 bytes from Logical Block Address 0 (sector 0, head 0, cylinder 0) of the device. This is always the first sector.
  2. It loads these 512 bytes into physical memory at address 0x00007C00.
  3. It checks the last two bytes of the loaded sector. They must be 0x55 followed by 0xAA (the "boot signature"). On little-endian x86, this appears as the 16-bit word 0xAA55.
  4. If the signature is valid, the BIOS performs a far jump to 0x0000:0x7C00, setting CS:IP = 0x0000:0x7C00, and your code begins executing.
  5. If the signature is invalid, the BIOS tries the next device in the boot order.

This means your boot sector must be exactly 512 bytes, and bytes 510-511 must contain the boot signature. This is non-negotiable.

1.2.4 Memory Layout at Boot Time

When your boot sector receives control, the physical memory layout looks like this:

; Physical Address Contents Size ; ────────────────────────────────────────────────────────────────── ; 0x00000 - 0x003FF Interrupt Vector Table (IVT) 1,024 bytes ; 0x00400 - 0x004FF BIOS Data Area (BDA) 256 bytes ; 0x00500 - 0x07BFF Free conventional memory ~30 KB ; 0x07C00 - 0x07DFF YOUR BOOT SECTOR (loaded here) 512 bytes ; 0x07E00 - 0x9FFFF Free conventional memory ~600 KB ; 0xA0000 - 0xAFFFF VGA Graphics Memory (Mode 13h) 64 KB ; 0xB0000 - 0xB7FFF Monochrome Text Buffer 32 KB ; 0xB8000 - 0xBFFFF Color Text Buffer (80x25) 32 KB ; 0xC0000 - 0xC7FFF Video Card ROM 32 KB ; 0xC8000 - 0xEFFFF Expansion ROM / Optional 160 KB ; 0xF0000 - 0xFFFFF Motherboard BIOS ROM 64 KB ; ────────────────────────────────────────────────────────────────── ; Total addressable in real mode: 1 MB (20 address lines: A0-A19)

The IVT occupies the first 1 KB of memory. It contains 256 entries, each 4 bytes, for a total of 1024 bytes. Each entry is a far pointer (segment:offset) to an interrupt handler. When the CPU executes INT 0x10, it looks up entry 0x10 in the IVT, finds the address at offset 0x0040, and jumps there. The BIOS has already filled the IVT with pointers to its own handler routines.

Critical Warning
The area from 0x00500 to 0x07BFF (just below your boot sector) and from 0x07E00 to 0x9FFFF (just above your boot sector) is free for your use. However, you must be careful not to overwrite the IVT, BDA, or video memory unless you know exactly what you are doing. Setting up your stack pointer to grow down from 0x7C00 is the standard practice, since the stack grows downward and the region below 0x7C00 is free.

1.3 Your First Boot Sector: "Hello, OS World!"

We now write our first complete boot sector. This program prints the message "Hello, OS World!" to the screen using BIOS interrupt 0x10 (video services), then halts the CPU. Every single line will be explained in the next section.

; ═══════════════════════════════════════════════════════════════════════════════
; boot.asm  —  Hello World Boot Sector
; Assemble:  nasm -f bin boot.asm -o boot.bin
; Test:      qemu-system-i386 -drive format=raw,file=boot.bin
; ═══════════════════════════════════════════════════════════════════════════════

[BITS 16]                   ; Assemble for 16-bit real mode
[ORG 0x7C00]                ; Code will be loaded at 0x7C00

start:
    cli                     ; Disable hardware interrupts
    xor ax, ax            ; AX = 0 (fastest way to zero a register)
    mov ds, ax            ; Data Segment = 0x0000
    mov es, ax            ; Extra Segment = 0x0000
    mov ss, ax            ; Stack Segment = 0x0000
    mov sp, 0x7C00        ; Stack Pointer = 0x7C00 (grows downward)
    sti                     ; Re-enable interrupts (stack is valid now)

    ; ── Print message ────────────────────────────────────────────────────
    mov si, msg           ; SI = pointer to message string

.print_loop:
    lodsb                   ; AL = [DS:SI], then SI = SI + 1
    test al, al           ; Is AL == 0 (null terminator)?
    jz .done             ; Yes → jump to .done
    mov ah, 0x0E          ; BIOS function: Teletype Output
    mov bh, 0x00          ; Page number 0
    mov bl, 0x0F          ; Colour: bright white (0x0F) on black
    int 0x10                ; Call BIOS video interrupt
    jmp .print_loop       ; Repeat for next character

.done:
    hlt                     ; Halt the CPU
    jmp .done             ; If NMI wakes us, halt again (safety)

; ── Data Section ──────────────────────────────────────────────────────
msg:
    db 'Hello, OS World!', 0x0D, 0x0A, 0
    ; 0x0D = Carriage Return, 0x0A = Line Feed, 0 = Null terminator

; ── Padding and Boot Signature ────────────────────────────────────────
times 510 - ($ - $$) db 0   ; Pad with zeros to byte 510
dw 0xAA55                  ; Boot signature at bytes 510-511

1.4 Line-by-Line Explanation

Every line of the boot sector is explained here in exhaustive detail. Understanding these fundamentals is absolutely essential before proceeding.

[BITS 16] — Assembler Mode Directive

This is not a CPU instruction. It is an assembler directive that tells NASM to generate 16-bit machine code. When the CPU first powers on, it starts in Real Mode, a compatibility mode dating back to the 8086 processor. In Real Mode, all general-purpose registers are 16 bits wide (AX, BX, CX, DX, etc.), memory addressing uses the segment:offset model, and the CPU can only directly access 1 MB of physical memory. The [BITS 16] directive ensures that NASM encodes instructions with 16-bit operand sizes and addressing modes.

[ORG 0x7C00] — Origin Directive

This directive sets the origin point for address calculations. NASM uses this value to compute the actual memory addresses of all labels. When you write mov si, msg, NASM must encode the address of msg into the instruction. If [ORG 0x7C00] is set, and msg is 30 bytes after the start of the file, NASM calculates its address as 0x7C00 + 30 = 0x7C1E. Without this directive, NASM would assume the code starts at address 0, and the label would have address 30 (0x001E). When the CPU executes the instruction at 0x7C00, it would try to read from address 0x001E (in the IVT!), which would produce garbage on screen instead of your message.

cli — Clear Interrupt Flag

This instruction clears the Interrupt Flag (IF) in the EFLAGS register. When IF = 0, maskable hardware interrupts are disabled. This means the timer (IRQ0), keyboard (IRQ1), and other hardware devices cannot interrupt the CPU. We disable interrupts because we are about to modify the stack pointer (SS:SP), and if an interrupt occurred between setting SS and setting SP, the CPU would push the return address (FLAGS, CS, IP) onto a stack with an invalid pointer, corrupting memory. Once SS:SP are both valid, we re-enable interrupts with sti.

xor ax, ax — XOR Register with Itself

This is the fastest and smallest way to set a register to zero. The XOR of any value with itself is always zero. This instruction encodes to just 2 bytes: 31 C0. The alternative, mov ax, 0, encodes to 3 bytes: B8 00 00. This pattern — XOR reg, reg — is used extensively in both hand-written and compiler-generated assembly. It also has the side effect of clearing the Carry Flag (CF) and Overflow Flag (OF), which is sometimes useful.

mov ds, ax — Load Data Segment Register

This sets the Data Segment (DS) register to 0. Segment registers cannot be loaded directly with immediate values in x86. You cannot write mov ds, 0 — the CPU simply does not support that encoding. You must load a general-purpose register first, then move it to the segment register. With DS = 0, any data access like mov al, [si] or lodsb will use physical address 0x00000 + SI. This is correct because our string is at 0x7C00+ (within the boot sector loaded there), and with DS=0 and SI pointing to the correct offset, the physical address is correct.

mov ss, ax / mov sp, 0x7C00 — Stack Setup

These two instructions set up the stack. SS:SP forms the stack pointer. SS = 0x0000, SP = 0x7C00. The x86 stack grows downward — every PUSH instruction first decrements SP by the operand size (2 bytes in 16-bit mode), then writes the value. So the first PUSH AX after this setup would write to address 0x7BFE (SP becomes 0x7BFE, then the value is stored). This is safe because the memory from 0x00500 to 0x07BFF is free. The stack can grow down to 0x0500 before it would overwrite the BDA, giving us about 30 KB of stack space — more than enough for a boot sector.

sti — Set Interrupt Flag

Re-enables hardware interrupts. Now that our stack is properly configured, interrupts can safely occur. If the timer fires, the CPU will push FLAGS, CS, and IP onto our valid stack and jump to the timer handler.

mov si, msg — Load String Pointer

Loads the 16-bit offset of the label msg into the SI (Source Index) register. Because of [ORG 0x7C00], NASM has computed the correct offset. If msg is at file offset 0x0020, SI will be loaded with 0x7C20. The instruction mov si, msg is encoded as a 16-bit immediate move. NASM resolves msg to its absolute address based on the origin.

lodsb — Load String Byte

This is one of the most elegant instructions in x86. It performs three operations atomically: AL = byte at [DS:SI], then SI = SI + 1 (if DF=0, which is the default). The entire string-loading loop body is just 4 instructions: lodsb, test al, al, jz .done, and the output code. Without lodsb, you would need: mov al, [si], inc si, which is two instructions and potentially slower.

test al, al — Test for Zero

Performs a bitwise AND of AL with itself, discards the result, and sets the flags. If AL is 0x00 (our null terminator), the Zero Flag (ZF) is set to 1. Unlike cmp al, 0, TEST does not require an immediate operand byte, saving space. It also does not modify AL, unlike AND.

jz .done — Jump if Zero

Conditional jump that transfers execution to .done if ZF = 1. This happens when we have reached the null byte at the end of the string. The label .done is a local label in NASM — the leading dot makes it local to the enclosing non-local label (start in this case). This helps prevent name collisions in large projects.

mov ah, 0x0E — BIOS Function Selection

Sets AH to 0x0E, which selects the Teletype Output sub-function of BIOS interrupt 0x10. The INT 0x10 handler uses AH to determine which operation to perform. AL contains the character to print, BH the display page, and BL the color attribute (in graphics modes). This function advances the cursor automatically and handles scrolling when the cursor reaches the bottom of the screen.

int 0x10 — Software Interrupt

Transfers control to the BIOS video interrupt handler. Here's exactly what happens inside the CPU: (1) Push FLAGS onto the stack, (2) Clear IF and TF in FLAGS, (3) Push CS onto the stack, (4) Push IP onto the stack, (5) Look up the handler address from the IVT at address 0x0000 + (0x10 × 4) = 0x0040, (6) Load CS:IP from the IVT entry, (7) Begin executing the BIOS handler. When the handler finishes with iret, it pops IP, CS, and FLAGS, returning control to our code.

times 510 - ($ - $$) db 0 — Padding Directive

This is a NASM-specific directive that fills bytes. $ represents the current assembly position, and $$ represents the start of the current section (0x7C00 due to ORG). So $ - $$ computes how many bytes have been emitted so far. 510 - ($ - $$) computes how many bytes remain before byte 510. db 0 emits a zero byte. times N db 0 repeats this N times. This ensures the boot signature ends up exactly at bytes 510-511 regardless of how much code and data we have.

dw 0xAA55 — Boot Signature

Emits a 16-bit word. On x86 (little-endian), the low byte (0x55) is stored at the lower address (byte 510), and the high byte (0xAA) is stored at the higher address (byte 511). This matches exactly what the BIOS expects: byte 510 = 0x55, byte 511 = 0xAA.

1.5.2 Assembling with NASM

$ nasm -f bin boot.asm -o boot.bin

The -f bin option is absolutely critical. It tells NASM to produce a flat binary — raw machine code with no file format headers. Other formats like elf32 or win32 would add metadata headers that the BIOS does not understand. The BIOS expects pure machine code starting at byte 0. Verify your output:

$ ls -la boot.bin

# Inspect the hex dump

Press Ctrl+Alt+G to release the mouse from QEMU's grab. Always use -no-reboot during development — if your code causes a triple fault (three consecutive exceptions), the CPU normally resets, causing an infinite reboot loop. The -no-reboot flag makes QEMU halt instead, letting you examine the state.

Mastering the language of the CPU: registers, memory, instructions, and flags

2.1 The x86 Register Architecture

The 8086 processor, released by Intel in 1978 and the ancestor of every modern x86 CPU, provided 14 programmer-visible registers. Understanding every one of these registers in detail is the single most important foundation for all assembly programming. Every x86 CPU from the 8086 through the modern Core i9 still supports these same registers (with extensions).

2.1.1 General-Purpose Registers

The four general-purpose registers — AX, BX, CX, DX — are 16 bits wide in real mode. Each can be accessed as two independent 8-bit halves: the high byte (H suffix) and the low byte (L suffix).

BXBHBLBaseMemory addressing as base pointer [BX+SI], [BX+DI]; BIOS parameter passing

32-Bit Extensions (386+)

When Intel introduced the 80386 in 1985, all registers were extended to 32 bits by prefixing with "E" (Extended): EAX, EBX, ECX, EDX, ESI, EDI, ESP, EBP. The lower 16 bits remain fully accessible as AX, BX, etc., and the lower bytes as AH/AL, BH/BL, etc. For our OS, we work in 32-bit protected mode, so we primarily use the E-prefixed registers.

; Register layout for EAX (same pattern applies to EBX, ECX, EDX):
;
;  31                              16 15       8 7        0
;   +--------------------------------+----------+----------+
;   |        EAX (32 bits)           │    AH    │    AL    │
;   +--------------------------------+----------+----------+
;   │              AX (16 bits)                   │
;
; Examples:
mov  eax, 0x12345678     ; EAX = 0x12345678
mov  al, 0xFF            ; AL = 0xFF → EAX becomes 0x123456FF
mov  ah, 0xAA            ; AH = 0xAA → EAX becomes 0x1234AAFF
mov  ax, 0xBEEF          ; AX = 0xBEEF → EAX becomes 0x1234BEEF
xor  eax, eax            ; EAX = 0x00000000 (clears entire register)
movzx eax, bl            ; EAX = zero-extended BL (upper 24 bits cleared)
movsx eax, bl            ; EAX = sign-extended BL (upper 24 bits = bit 7 of BL)

Segment registers hold the base address for memory segments. In real mode, they are shifted left 4 bits and added to the offset. In protected mode, they contain selectors — indices into the GDT.

2.1.4 The Instruction Pointer and Flags

IP / EIP: The Instruction Pointer holds the offset of the next instruction to execute within the code segment (CS:IP or CS:EIP). You cannot directly modify IP with a MOV instruction. It changes automatically as instructions execute, and is modified by control flow instructions (JMP, CALL, RET, INT, IRET, conditional jumps). The only way to indirectly set IP is through these control flow instructions or by manually manipulating the stack and executing RET or IRET.

FLAGS / EFLAGS: A 32-bit register containing status and control bits. Individual flags are used for conditional branching, interrupt control, and CPU operation modes. We will explore each flag in Section 2.4.

2.2 Memory Addressing Modes

x86 provides extraordinary flexibility in how memory operands are specified. Understanding every addressing mode is essential for efficient assembly programming.

2.2.1 Direct Addressing

mov  ax, [0x7C00]          ; Load 16-bit word from DS:0x7C00
mov  al, [my_variable]     ; Load byte from address of my_variable
mov  [es:0xB8000], ax      ; Store to ES:0xB8000 (VGA memory, with segment override)

2.2.2 Register Indirect Addressing

mov  ax, [bx]              ; Load from DS:BX
mov  [si], al              ; Store AL to DS:SI
mov  ax, [ebp]             ; Load from SS:EBP (BP/EBP default to SS!)
mov  ax, [esp]             ; Load from SS:ESP

2.2.3 Based Addressing

mov  ax, [bx + 4]          ; Load from DS:BX+4
mov  ax, [bp - 8]          ; Load from SS:BP-8 (local variable access)
mov  al, [esi + 10]        ; Load from DS:ESI+10

2.2.4 Indexed Addressing

mov  ax, [si + di]         ; Invalid! Cannot combine two index registers this way
mov  ax, [bx + si]         ; Valid: base + index
mov  ax, [bx + di]         ; Valid: base + index
mov  ax, [bp + si]         ; Valid: base + index (defaults to SS)
mov  ax, [bp + di]         ; Valid: base + index (defaults to SS)

2.2.5 Based-Indexed with Displacement

; Most complex addressing mode - base + index + constant offset
mov  ax, [bx + si + 4]     ; DS:BX+SI+4
mov  ax, [bp + di - 8]     ; SS:BP+DI-8
; This is how C compilers access struct members and array elements

2.2.6 Scaled Indexed Addressing (386+ Only)

; [base + index*scale + displacement] - 32-bit only
mov  eax, [ebx + esi*4]        ; DS:EBX + ESI*4 (perfect for 32-bit arrays)
mov  eax, [array + ecx*8 + 4]  ; DS:array + ECX*8 + 4
; Scale can be 1, 2, 4, or 8 only
Addressing Mode Encoding
Not all combinations are valid. The 16-bit addressing modes are severely limited: you can only use BX or BP as the base, and SI or DI as the index. BX and BP cannot both appear, nor can SI and DI both appear. In 32-bit mode, any general-purpose register except ESP can be the index, and any register can be the base.

2.3 The Instruction Set: A Comprehensive Reference

2.3.1 Data Transfer Instructions

MOV dst, src        — Move (copy) src to dst. Does NOT affect flags.
                     Forms: reg↔reg, reg↔mem, mem↔reg, imm→reg, imm→mem
                     Cannot: mem↔mem directly, imm→seg, seg→seg

  mov  ax, bx          ; AX = BX
  mov  [0x1000], al    ; Store AL to DS:0x1000
  mov  dx, 0x3D4       ; DX = 0x3D4 (immediate to register)
  mov  eax, cr3        ; Read control register (privileged)
  mov  cr3, eax        ; Write control register (privileged)

MOVZX dst, src      — Move with Zero-Extend. 386+ only.
                     Copies smaller src to larger dst, zeroing upper bits.
  movzx eax, al        ; EAX = 000000XX (AL zero-extended to 32 bits)
  movzx eax, word [x]  ; EAX = 0000XXXX (16-bit word zero-extended)

MOVSX dst, src      — Move with Sign-Extend. 386+ only.
                     Copies smaller src to larger dst, replicating sign bit.
  movsx eax, al        ; If AL=0x80, EAX=0xFFFFFF80 (negative preserved)
  movsx eax, bl        ; If BL=0x7F, EAX=0x0000007F (positive preserved)

XCHG dst, src       — Exchange. Swaps two values atomically.
  xchg ax, bx          ; Swap AX and BX
  xchg eax, [mutex]    ; Atomic swap (has implicit LOCK prefix with memory)

LEA dst, src        — Load Effective Address. Computes address, does NOT access memory.
  lea  eax, [ebx+ecx*4+8]  ; EAX = EBX + ECX*4 + 8 (faster than ADD+SHL+ADD)
  lea  esi, [my_array]     ; ESI = address of my_array (same as MOV for simple labels)

PUSH src            — Push onto stack. ESP = ESP - 4 (32-bit) or -2 (16-bit), then store.
  push ax              ; ESP = ESP - 2, [SS:ESP] = AX
  push eax             ; ESP = ESP - 4, [SS:ESP] = EAX
  pushfd               ; Push EFLAGS register
  push 0x7C00          ; Push immediate value

POP dst             — Pop from stack. Load from [SS:ESP], then ESP = ESP + size.
  pop  ax              ; AX = [SS:ESP], ESP = ESP + 2
  pop  eax             ; EAX = [SS:ESP], ESP = ESP + 4
  popfd                ; Restore EFLAGS from stack

PUSHA / POPA        — Push/Pop All (16-bit). Pushes AX,CX,DX,BX,SP,BP,SI,DI.
PUSHAD / POPAD      — Push/Pop All (32-bit). Pushes EAX,ECX,EDX,EBX,ESP,EBP,ESI,EDI.

2.3.2 Arithmetic Instructions

ADD dst, src        — dst = dst + src. Sets CF, ZF, SF, OF, PF, AF.
  add  ax, 5           ; AX = AX + 5
  add  [counter], byte 1  ; Increment byte at [counter]

ADC dst, src        — Add with Carry. dst = dst + src + CF.
  adc  eax, ebx        ; Used for multi-precision arithmetic (e.g., 64-bit addition)

SUB dst, src        — dst = dst - src. Sets flags (CF=borrow).
  sub  sp, 16          ; Allocate 16 bytes on stack
  sub  eax, [total]    ; EAX = EAX - value at [total]

SBB dst, src        — Subtract with Borrow. dst = dst - src - CF.
  sbb  edx, ecx        ; Used for multi-precision subtraction

INC dst             — Increment by 1. Sets all flags EXCEPT CF.
  inc  eax             ; EAX = EAX + 1 (CF preserved - useful in loops using carry)

DEC dst             — Decrement by 1. Sets all flags EXCEPT CF.
  dec  ecx             ; ECX = ECX - 1

NEG dst             — Negate (two's complement). dst = 0 - dst.
  neg  eax             ; EAX = -EAX

CMP dst, src        — Compare. Computes dst - src, discards result, sets flags.
  cmp  ax, 10          ; Sets ZF if AX==10, sets CF if AX<10 (unsigned)
  cmp  eax, [limit]    ; Compare EAX with memory value

MUL src             — Unsigned multiply.
  mul  bl              ; AX = AL * BL (8-bit)
  mul  bx              ; DX:AX = AX * BX (16-bit, result in two registers)
  mul  ebx             ; EDX:EAX = EAX * EBX (32-bit)

IMUL src            — Signed multiply. Same forms as MUL.
  imul ebx             ; EDX:EAX = EAX * EBX (signed)
  imul eax, ebx, 10    ; EAX = EBX * 10 (3-operand form, 386+)

DIV src             — Unsigned divide.
  div  bl              ; AL = AX / BL (quotient), AH = AX % BL (remainder)
  div  bx              ; AX = DX:AX / BX, DX = DX:AX % BX
  div  ebx             ; EAX = EDX:EAX / EBX, EDX = EDX:EAX % EBX

IDIV src            — Signed divide. Same form as DIV.

2.3.3 Logical and Bitwise Instructions

AND dst, src        — Bitwise AND. dst = dst & src. Clears CF, OF.
  and  al, 0x0F        ; Mask off upper 4 bits (isolate hex digit)
  and  eax, 0xFFFFF000 ; Mask to 4KB page boundary

OR dst, src         — Bitwise OR. dst = dst | src. Sets individual bits.
  or   al, 0x20        ; Set bit 5
  or   eax, 0x80000000 ; Set PG bit in CR0

XOR dst, src        — Bitwise XOR. dst = dst ^ src. Clears CF, OF.
  xor  eax, eax        ; Zero register (2 bytes, faster than MOV EAX,0)
  xor  eax, 0xFFFFFFFF ; Bitwise NOT (one's complement)

NOT dst             — Bitwise NOT. dst = ~dst. Does NOT affect flags.
  not  eax             ; Flip all 32 bits

TEST dst, src       — AND but discard result. Sets flags only.
  test al, al          ; ZF=1 if AL==0 (used before JZ/JNZ)
  test eax, 0x80000000 ; SF=1 if bit 31 is set

SHL dst, count      — Shift Left Logical. MSB goes to CF, LSB becomes 0.
  shl  ax, 1           ; AX = AX * 2, CF = old bit 15
  shl  eax, cl         ; EAX shifted left by CL bits

SHR dst, count      — Shift Right Logical. LSB goes to CF, MSB becomes 0.
  shr  ax, 1           ; AX = AX / 2 (unsigned)

SAR dst, count      — Shift Arithmetic Right. LSB to CF, MSB replicated.
  sar  ax, 1           ; AX = AX / 2 (signed, preserves sign)

ROL/ROR dst, count  — Rotate Left/Right. Bits wrap around.
  rol  al, 1           ; Rotate left by 1 (old MSB becomes new LSB and CF)

BT/BTS/BTR/BTC      — Bit Test/Test-and-Set/Test-and-Reset/Test-and-Complement. 386+.
  bt   eax, 5          ; CF = bit 5 of EAX
  bts  eax, 5          ; CF = bit 5, then set bit 5

2.3.4 Control Flow Instructions

JMP target          — Unconditional jump. Near, short, or far.
  jmp  .label          ; Short/near jump to label
  jmp  0x08:0x100000   ; Far jump: new CS=0x08, new EIP=0x100000

JE/JZ label         — Jump if Equal / Zero. ZF=1.
JNE/JNZ label       — Jump if Not Equal / Not Zero. ZF=0.
JG/JNLE label       — Jump if Greater (signed). ZF=0 and SF=OF.
JGE/JNL label       — Jump if Greater or Equal (signed). SF=OF.
JL/JNGE label       — Jump if Less (signed). SF≠OF.
JLE/JNG label       — Jump if Less or Equal (signed). ZF=1 or SF≠OF.
JA/JNBE label       — Jump if Above (unsigned). CF=0 and ZF=0.
JAE/JNB label       — Jump if Above or Equal (unsigned). CF=0.
JB/JC/JNAE label    — Jump if Below / Carry (unsigned). CF=1.
JBE/JNA label       — Jump if Below or Equal (unsigned). CF=1 or ZF=1.
JO/JNO, JS/JNS, JP/JNP — Jump on Overflow, Sign, Parity.
JCXZ label          — Jump if CX=0. JECXZ label — Jump if ECX=0 (386+).

CALL target         — Call subroutine. Pushes return address, then jumps.
  call print_string    ; Push IP, jump to print_string
  call eax             ; Indirect call through register
  call [func_table]    ; Indirect call through memory

RET [n]             — Return from subroutine. Pops return address into IP.
  ret                  ; Simple return
  ret  8               ; Return and add 8 to SP (discard 8 bytes of parameters)

INT n               — Software interrupt. Pushes FLAGS, CS, IP; jumps to IDT[n].
  int  0x10            ; BIOS video interrupt
  int  0x80            ; Linux system call (we'll implement our own)

IRET                — Interrupt Return. Pops IP, CS, FLAGS. Used by all interrupt handlers.

LOOP label          — Decrement ECX (or CX), jump if not zero.
  mov  ecx, 10
.next:
  ; ... body ...
  loop .next           ; ECX--, jump if ECX != 0

LOOPE/LOOPZ label   — Loop while Equal/Zero. ECX--, jump if ECX≠0 and ZF=1.
LOOPNE/LOOPNZ label — Loop while Not Equal/Not Zero. ECX--, jump if ECX≠0 and ZF=0.

CALL/RET using the stack:
  ; These are equivalent:
  call some_func       ; Push return address, jump
  ; ...
  some_func:
      ; ... do work ...
      ret              ; Pop return address, jump back

  ; Manual equivalent:
  push .return_here
  jmp  some_func
.return_here:
  ; ...
  some_func:
      ; ... do work ...
      pop  eax         ; Get return address
  jmp  eax           ; Jump to it

2.5 Stack Operations in Depth

The stack is the most important data structure in low-level programming. It is used for function calls, local variables, parameter passing, interrupt handling, and temporary storage.

2.5.1 Stack Mechanics

; The stack grows DOWNWARD in memory. SS:SP points to the last pushed item.
; On 32-bit: PUSH decrements ESP by 4, stores 4 bytes.
; On 16-bit: PUSH decrements SP by 2, stores 2 bytes.

; Initial state: ESP = 0x90000
push  eax        ; ESP = 0x8FFFC, [0x8FFFC] = EAX
push  ebx        ; ESP = 0x8FFF8, [0x8FFF8] = EBX
push  ecx        ; ESP = 0x8FFF4, [0x8FFF4] = ECX

pop   edx        ; EDX = ECX, ESP = 0x8FFF8
pop   eax        ; EAX = EBX, ESP = 0x8FFFC

; Stack frame setup (standard function prologue):
push  ebp        ; Save caller's frame pointer
mov   ebp, esp   ; Set up new frame pointer
sub   esp, 16    ; Allocate 16 bytes of local variables

; Access local variables (negative offsets from EBP):
mov   [ebp - 4], eax    ; local_var_1 = EAX
mov   ebx, [ebp - 8]    ; EBX = local_var_2

; Access parameters (positive offsets from EBP, +4 is return address):
mov   eax, [ebp + 8]    ; First parameter
mov   ebx, [ebp + 12]   ; Second parameter

; Function epilogue:
mov   esp, ebp   ; Deallocate locals
pop   ebp        ; Restore caller's frame pointer
ret              ; Return

2.6 String Instructions

String instructions are among the most powerful and efficient in x86. They operate on sequences of bytes, words, or doublewords, automatically advancing pointers. They are used for memory copying, zeroing, comparing, and scanning.

; All string instructions use:
;   DS:SI (or DS:ESI) as source
;   ES:DI (or ES:EDI) as destination
;   Direction Flag (DF) controls increment (DF=0) or decrement (DF=1)

CLD                    ; Clear Direction Flag: SI/DI increment after each operation
STD                    ; Set Direction Flag: SI/DI decrement after each operation

MOVSB / MOVSW / MOVSD  ; Move string byte/word/dword
; [ES:DI] = [DS:SI], then SI+=size, DI+=size
rep movsd              ; Copy ECX dwords from DS:ESI to ES:EDI
; (Used for fast memory copy)

STOSB / STOSW / STOSD  ; Store string: [ES:DI] = AL/AX/EAX, advance DI
cld
xor   eax, eax         ; EAX = 0
mov   ecx, 1024        ; 1024 dwords = 4096 bytes
rep   stosd            ; Zero 4096 bytes starting at ES:EDI
; (Fastest way to zero memory)

LODSB / LODSW / LODSD  ; Load string: AL/AX/EAX = [DS:SI], advance SI
; Usually used inside loops, not with REP

CMPSB / CMPSW / CMPSD  ; Compare strings: [DS:SI] - [ES:DI], advance both
repe cmpsb             ; Compare bytes while equal (ZF=1) and ECX≠0
; (Implements memcmp)

SCASB / SCASW / SCASD  ; Scan string: AL/AX/EAX - [ES:DI], advance DI
repne scasb            ; Scan for byte AL while not equal (ZF=0) and ECX≠0
; (Implements strchr / strlen)

3.1 NASM Syntax Philosophy

NASM (Netwide Assembler) follows an Intel-syntax approach that is clean, consistent, and explicit. Unlike GAS (GNU Assembler) which uses AT&T syntax (e.g., movl $5, %eax), NASM uses Intel syntax where the destination comes first: mov eax, 5. NASM is designed to be straightforward: what you write is what you get, with no hidden instruction transformations.

Key principles of NASM syntax: (1) Square brackets [] always mean "memory access." mov ax, bx copies a register, while mov ax, [bx] reads from memory. (2) The size of memory operands is never implicit — you must specify byte, word, or dword when the size cannot be determined from the other operand. (3) Labels are defined by writing their name followed by a colon. Labels with a leading dot are local to the previous non-local label.

; NASM vs GAS (AT&T) syntax comparison:
; Operation              NASM (Intel)              GAS (AT&T)
; ──────────────────────────────────────────────────────────────────
; Register move          mov  eax, ebx             movl %ebx, %eax
; Immediate value        mov  eax, 42              movl $42, %eax
; Memory read            mov  eax, [ebx]           movl (%ebx), %eax
; Memory with offset     mov  eax, [ebx+4]         movl 4(%ebx), %eax
; Indexed access         mov  eax, [ebx+esi*2]     movl (%ebx,%esi,2), %eax
; Immediate to memory    mov  [addr], dword 5      movl $5, addr
; Size override          mov  ax, [byte ptr]       movb addr, %al

3.2 Directives Comprehensive Reference

NASM directives (also called pseudo-instructions) control the assembly process. They are not translated into machine code.

3.2.1 Mode and Origin Directives

[BITS 16]              ; Generate 16-bit code (real mode)
[BITS 32]              ; Generate 32-bit code (protected mode)
[BITS 64]              ; Generate 64-bit code (long mode)
[ORG 0x7C00]           ; Set origin point for address calculations
[ORG 0x100000]         ; Kernel loaded at 1 MB

The [BITS] directive affects instruction encoding. In 16-bit mode, mov ax, [bx] is encoded naturally. In 32-bit mode, the same instruction would need an operand-size override prefix to use 16-bit registers. NASM automatically handles these prefixes based on the BITS setting and the registers you use.

3.2.2 Section Directives

section .text          ; Code section (executable, read-only)
section .data          ; Initialized data section (readable, writable)
section .bss           ; Uninitialized data section (zero-filled at load)
section .rodata        ; Read-only data (string literals, constants)

; For flat binary output (-f bin), sections are concatenated in order.
; For ELF output (-f elf32), sections are placed by the linker.

; Example of a properly structured assembly file:
section .text
    global _start
_start:
    call main
    ; ...

section .data
hello:    db "Hello", 0
counter:  dd 42              ; dword (4 bytes) initialized to 42

section .bss
buffer:   resb 1024         ; Reserve 1024 bytes (uninitialized)
array:    resd 256          ; Reserve 256 dwords (1024 bytes total)

3.2.3 Data Definition Directives

DB  (Define Byte)      — Emit one or more bytes
  db  0x55               ; Single byte
  db  'A'                ; ASCII character (0x41)
  db  'Hello', 0         ; String with null terminator
  db  1, 2, 3, 4, 5      ; Five bytes

DW  (Define Word)      — Emit one or more 16-bit words (little-endian)
  dw  0xAA55             ; Bytes: 55 AA (little-endian)
  dw  1000               ; Word value 1000 (0x03E8)

DD  (Define Dword)     — Emit one or more 32-bit doublewords
  dd  0x12345678         ; Bytes: 78 56 34 12
  dd  0x00100000         ; Physical address 1 MB

DQ  (Define Qword)     — Emit 64-bit quadwords
  dq  0x123456789ABCDEF0

RESB (Reserve Bytes)   — Reserve uninitialized bytes (BSS only)
  resb 512               ; Reserve 512 bytes (filled with zeros by loader)

RESW, RESD, RESQ       — Reserve words, dwords, qwords

3.2.4 The TIMES Directive

TIMES is one of NASM's most powerful features. It repeats an instruction or data declaration a specified number of times.

times 510 - ($ - $$) db 0    ; Pad to byte 510 (boot sector)
times 256 dd 0                ; Reserve 256 dwords (1 KB of zeros)
times 80 db ' '               ; 80 space characters
times 100 nop                 ; 100 NOP instructions (code alignment or delay)

The expression $ - $$ is fundamental: $ is the current assembly position, and $$ is the start of the current section. Their difference is the number of bytes emitted so far in this section.

Chapter 4
Real Mode Programming — Segmentation, BIOS, and Memory

Mastering 16-bit x86 before the transition to protected mode

4.1 Real Mode Architecture

Real Mode is the operating mode of the x86 CPU immediately after reset. It is a 16-bit environment that mimics the original 8086 processor while maintaining backward compatibility with all later x86 CPUs. In Real Mode: all registers are 16 bits wide (except the 32-bit extensions which are accessible with override prefixes); memory addresses are 20 bits, allowing access to 1 MB of physical memory (0x00000 to 0xFFFFF); memory uses the segmented addressing model; there is no memory protection — any code can access any memory and any I/O port; there are no privilege levels (no Ring 0 vs Ring 3); and interrupts are handled through the Interrupt Vector Table (IVT) at the bottom of memory.

Real Mode is where every x86 OS begins. Even UEFI systems start in a compatibility mode before transitioning. Understanding Real Mode thoroughly is non-negotiable for OS developers.

4.2 Segmentation in Depth

In Real Mode, physical addresses are calculated using the formula: Physical Address = (Segment × 16) + Offset. The segment value is shifted left by 4 bits (equivalent to multiplying by 16 or adding a hex digit) and then added to the 16-bit offset.

; Physical address calculation examples:
; Segment = 0x0000, Offset = 0x7C00  →  Physical = 0x00000 + 0x7C00 = 0x07C00
; Segment = 0x07C0, Offset = 0x0000  →  Physical = 0x07C00 + 0x0000 = 0x07C00
; Segment = 0xFFFF, Offset = 0x000F  →  Physical = 0xFFFF0 + 0x000F = 0xFFFF
;                                      (wraps around to 0xFFFF on 8086, 0x0FFFF on 286+ with A20)

; Loading segment registers:
mov  ax, 0xB800
mov  es, ax          ; ES = 0xB800 (video memory segment)
; Now [ES:0] accesses physical address 0xB8000
mov  ax, [es:0]      ; Read from 0xB8000

; Segment overrides:
mov  ax, [cs:bx]     ; Use Code Segment for data access
mov  ax, [ss:si]     ; Use Stack Segment (default for BP-based anyway)
lodsb                ; Reads from DS:SI
movsb                ; Reads from DS:SI, writes to ES:DI
Segment:Offset Aliasing
The same physical address can be represented by many different segment:offset combinations. This property, called "segment:offset aliasing," is a fundamental characteristic of Real Mode. For example, 0x07C00 can be expressed as 0x0000:0x7C00, 0x07C0:0x0000, 0x07B0:0x0100, or 0x0780:0x0400. This made memory management in Real Mode notoriously difficult.

The A20 Line & Memory Detection

Accessing memory above 1 MB and mapping the full RAM layout

5.1 The A20 Problem: A Historical Quirk

The original Intel 8086 processor had 20 address lines (A0 through A19), giving it exactly 1 MB of addressable memory (220 = 1,048,576 bytes). Programmers discovered that due to the segment:offset addressing, memory addresses "wrapped around" at the 1 MB boundary. For example, the address FFFF:0010 computes to physical address (0xFFFF × 16) + 0x0010 = 0xFFFF0 + 0x0010 = 0x100000, which is exactly 1 MB. But the 8086 only had 20 address lines, so bit 20 (the "1" in 0x100000) was discarded, and this address actually accessed physical location 0x00000.

Some early PC programs intentionally relied on this wrap-around behavior. When IBM designed the AT (80286-based) with 24 address lines, they needed to preserve this behavior for backward compatibility. Their solution was the A20 Gate — a hardware latch on the keyboard controller (Intel 8042) that could electrically disable the A20 address line. With A20 gated OFF, address 0x100000 appears as 0x00000, preserving the wrap-around. To access memory above 1 MB ("extended memory"), we must enable A20.

Why This Matters
Without the A20 line enabled, any address with bit 20 set will alias back to the corresponding address below 1 MB. When we transition to protected mode, we need to load the GDT and IDT above 1 MB. If A20 is disabled, those loads will corrupt the IVT and BDA, causing immediate system crashes that are extremely difficult to debug. Always enable A20 before entering protected mode.

5.2 Methods to Enable A20

5.2.1 Method 1: BIOS INT 0x15 Function 0x2401 (Recommended First Try)

enable_a20_bios:
    mov ax, 0x2401       ; AH=0x24 (A20 functions), AL=0x01 (enable)
    int 0x15
    jc .failed         ; CF=1 means function not supported
    ret
.failed:
    jmp enable_a20_kbc  ; Fall through to keyboard controller method

; Check if A20 is enabled (function 0x2402):
check_a20_bios:
    mov ax, 0x2402
    int 0x15
    ; Returns: AL = status (0=disabled, 1=enabled), CF on error

5.2.2 Method 2: Keyboard Controller (8042) — Universal but Slow

enable_a20_kbc:
    call kbc_wait_input
    mov al, 0xAD       ; Command: Disable keyboard
    out 0x64, al       ; Send to keyboard controller command port

    call kbc_wait_input
    mov al, 0xD0       ; Command: Read output port
    out 0x64, al

    call kbc_wait_output

    push ax            ; Save it

    out 0x64, al

    pop ax
    or al, 2           ; Set bit 1 (A20 enable bit)
    out 0x60, al       ; Write modified value

    call kbc_wait_input
    mov al, 0xAE       ; Command: Enable keyboard
    out 0x64, al

    ret

; Wait until keyboard controller input buffer is empty

    in al, 0x64        ; Read status register
    test al, 2         ; Bit 1: Input buffer full?
    jnz kbc_wait_input
    ret

; Wait until keyboard controller output buffer has data
kbc_wait_output:
    in al, 0x64        ; Read status register
    test al, 1         ; Bit 0: Output buffer full?
    jz kbc_wait_output
    ret

enable_a20_fast:
    in al, 0x92        ; Read System Control Port A
    test al, 2         ; Is A20 already enabled?
    jnz .done          ; If yes, skip
    or al, 2           ; Set A20 enable bit (bit 1)
    and al, 0xFE      ; CRITICAL: Clear bit 0 (reset bit!)
    out 0x92, al       ; Write back
.done:
    ret

Bit 0 of port 0x92 is connected to the CPU reset line. If you set it, the CPU immediately resets. You MUST mask it off with AND AL, 0xFE before writing, or your system will reboot the instant you try to enable A20. This is a classic bug that causes "infinite reboot loops" during development.



5.2.4 Verifying A20 is Actually Enabled


; Test if A20 is enabled by checking memory wrapping:
; Write a test value to 0x0000:0x0500 (physical 0x00500)
; Write a different value to 0xFFFF:0x0510
;   If A20 OFF: 0xFFFF:0x0510 = 0xFFFF0 + 0x0510 = 0x000500 (wraps)
;   If A20 ON:  0xFFFF:0x0510 = 0xFFFF0 + 0x0510 = 0x100500 (does NOT wrap)
; So if A20 is off, the write to 0xFFFF:0x0510 will overwrite 0x0000:0x0500!

check_a20:
    pushf
    cli                  ; No interrupts during test
    push ds
    push es
    push di
    push si

    xor ax, ax
    mov es, ax         ; ES = 0x0000
    mov di, 0x0500     ; ES:DI = 0x0000:0x0500

    mov ax, 0xFFFF
    mov ds, ax         ; DS = 0xFFFF
    mov si, 0x0510     ; DS:SI = 0xFFFF:0x0510

    mov al, [es:di]   ; Save original value at 0x00500
    push ax
    mov [es:di], byte 0x00  ; Write test value 0x00
    mov [ds:si], byte 0xFF  ; Write different value via wrap address

    cmp byte [es:di], 0xFF  ; Did the write alias?
    pop ax
    mov [es:di], al    ; Restore original value

    mov ax, 0
    je .exit           ; If aliased, A20 is disabled
    mov ax, 1          ; A20 is enabled!

.exit:
    pop si
    pop di
    pop es
    pop ds
    popf
    ret
    ; Returns AX=1 if enabled, AX=0 if disabled



5.3 E820 Memory Map Detection


Before entering protected mode, we must know which memory regions are actually usable. The BIOS INT 0x15, AX=0xE820 service is the definitive method. It returns a list of memory regions, each tagged with a type indicating whether it is usable RAM, reserved by hardware, or used for ACPI data.



; E820 Memory Map Entry Structure (20 bytes basic, 24 bytes with ACPI 3.0):
struc e820_entry
    .base    resq 1     ; 8 bytes: Base address of region
    .length  resq 1     ; 8 bytes: Length in bytes
    .type    resd 1     ; 4 bytes: Region type
endstruc

; Region types:
;   1 = Usable RAM (normal memory)
;   2 = Reserved (do NOT use — may be memory-mapped devices)
;   3 = ACPI Reclaimable (usable after ACPI tables are parsed)
;   4 = ACPI NVS (Non-Volatile Storage — must preserve across sleep)
;   5 = Bad memory (contains errors — do not use)
;   Any other = Treat as Reserved

get_memory_map:
    pusha
    mov di, 0x1004     ; Store entries at 0x1004 (leave room for count)
    xor ebx, ebx       ; EBX=0 means "first call"
    xor bp, bp         ; BP = entry count
    mov edx, 0x534D4150  ; Magic: "SMAP" in little-endian ASCII

.loop:
    mov eax, 0xE820
    mov ecx, 24         ; Request 24-byte entries (ACPI 3.0 extended)
    int 0x15

    jc .failed         ; CF=1 = error or unsupported
    cmp eax, 0x534D4150  ; BIOS must return "SMAP" in EAX
    jne .failed

    jcxz .skip         ; Skip zero-length entries
    cmp cl, 20         ; Got at least 20 bytes?
    jl .no_ext

.no_ext:
    cmp dword [es:di+16], 1  ; Check entry type
    jne .skip          ; Skip non-RAM entries for our simple map

    cmp dword [es:di+8], 0   ; Check length high dword
    jne .valid
    cmp dword [es:di+4], 0   ; Check length low dword
    je .skip          ; Skip zero-length entries

.valid:
    inc bp             ; Increment entry count
    add di, 24         ; Next entry slot

.skip:
    test ebx, ebx      ; EBX=0 means this was the last entry
    jnz .loop

    mov [0x1000], bp    ; Store entry count at 0x1000
    popa
    clc                  ; Success: clear carry flag
    ret

.failed:
    popa
    stc                  ; Failure: set carry flag
    ret



5.4 Building a Physical Memory Manager


With the E820 map stored, we can build a simple physical memory allocator. The strategy: parse the E820 entries, mark usable RAM pages (4 KB each), and implement a bitmap allocator.



; Simple physical page allocator (pre-paging, using E820 data):
; We maintain a bitmap where each bit represents one 4KB physical page.
; 0 = free, 1 = allocated. With 4GB of address space, we need 128KB of bitmap.

section .bss
page_bitmap:    resb 131072    ; 128 KB bitmap (covers 4 GB / 4096 pages)
total_pages:    resd 1
free_pages:     resd 1

section .text
; init_physical_memory: Parse E820 and build bitmap
; Input: E820 data at 0x1000 (count) and 0x1004 (entries)
init_physical_memory:
    pusha
    xor eax, eax
    mov edi, page_bitmap
    mov ecx, 32768      ; 128KB / 4 = 32768 dwords
    rep stosd            ; Zero the bitmap (all pages free)

    mov si, 0x1004     ; First E820 entry
    mov cx, [0x1000]  ; Entry count
    test cx, cx
    jz .done

.entry_loop:
    ; Mark low 1MB as allocated (reserved for BIOS)
    ; Mark any non-type-1 entries as allocated
    ; For now, simple approach: just track totals
    add si, 24
    dec cx
    jnz .entry_loop

.done:
    popa
    ret

6.1 Disk Geometry and Addressing


Hard disks and floppy disks organize data in concentric circles called tracks, which are divided into sectors. Multiple platters (disk surfaces) provide multiple heads (one per surface). A cylinder is the collection of all tracks at the same radius across all surfaces.





TermDefinition



CylinderAll tracks at the same radius across all platters. Numbered from 0 outward.


HeadA read/write head on a specific surface. Numbered from 0.


SectorA subdivision of a track. The smallest addressable unit, typically 512 bytes. Numbered from 1 (not 0!).


ClusterA group of contiguous sectors. The smallest allocation unit in a file system.






6.2 CHS vs LBA Addressing


CHS (Cylinder-Head-Sector) is the legacy addressing method. It requires three parameters and reflects physical disk geometry. However, BIOS INT 0x13 AH=0x02 can only access about 8 GB due to field size limits. LBA (Logical Block Addressing) treats the disk as a flat array of sectors numbered from 0. It is simpler and supports much larger drives.



; CHS to LBA conversion:
; LBA = ((Cylinder * NumHeads) + Head) * SectorsPerTrack + (Sector - 1)

; LBA to CHS conversion (needed for BIOS AH=0x02):
; temp   = LBA / SectorsPerTrack
; Sector = (LBA mod SectorsPerTrack) + 1
; Head   = temp mod NumHeads
; Cyl    = temp / NumHeads

lba_to_chs:
    push bx
    mov bx, ax         ; Save LBA in BX

    xor dx, dx
    div word [sectors_per_track]  ; AX = LBA / SPT, DX = LBA % SPT
    inc dx             ; Sector = remainder + 1 (sectors start at 1)
    mov cl, dl         ; CL = sector number

    xor dx, dx
    div word [num_heads]         ; AX = cylinder, DX = head
    mov dh, dl         ; DH = head
    mov ch, al         ; CH = cylinder low 8 bits

    mov al, ah
    and al, 0x03       ; Isolate cylinder bits 8-9
    shl al, 6          ; Shift to positions 6-7 of CL
    or cl, al          ; CL[7:6] = cylinder high bits, CL[5:0] = sector

    pop bx
    ret

sectors_per_track:    dw 18
num_heads:            dw 2

Chapter 7
Protected Mode — GDT, Descriptors, The Big Switch


Leaving real mode forever: entering the 32-bit world



7.1 Why Protected Mode?


Protected Mode, introduced with the 80286 and significantly enhanced in the 80386, transforms the x86 CPU from a simple 16-bit processor into a modern computing platform. In Protected Mode: registers are 32 bits wide (EAX, EBX, etc.); the CPU can address up to 4 GB of memory (232 bytes); memory is protected through segment limit checks and page-level permissions; four privilege levels (Ring 0-3) separate the kernel from user programs; hardware multitasking support is provided via the TSS; and virtual memory through paging allows each process to have its own address space.



The transition from Real Mode to Protected Mode is a one-way gate (without resetting the CPU). Once you enter Protected Mode, all memory accesses go through the GDT, and segment registers contain selectors, not raw segment bases.



7.2 The Global Descriptor Table (GDT)


The GDT is an array of 8-byte segment descriptors stored in memory. Each descriptor defines a memory segment: its base address, its limit (size), and its attributes (type, privilege level, granularity). The CPU references GDT entries through segment selectors — 16-bit values loaded into segment registers.



; GDT Descriptor Format (8 bytes = 64 bits):
;
;  Bit:  63      56 55 52 51 48 47    40 39      32 31      16 15       0
;       +---------+-----+-----+--------+----------+----------+----------+
;       │ Base    │Flags│Limit│ Access │ Base     │ Base     │ Limit    │
;       │[31:24]  │G D  │[19:16]│ Byte │ [23:16]  │ [15:0]   │ [15:0]   │
;       │         │B L  │     │ P D S  │          │          │          │
;       │         │0 0  │     │ L E T  │          │          │          │
;       +---------+-----+-----+--------+----------+----------+----------+
;        1 byte    1 nibble 1 nibble  1 byte     1 byte     2 bytes    2 bytes
;
; Access Byte (bits 47:40):
;   Bit 7 (P): Present. Must be 1 for valid descriptor.
;   Bits 6:5 (DPL): Descriptor Privilege Level (0=kernel, 3=user).
;   Bit 4 (S): 1=code/data segment, 0=system segment.
;   Bit 3 (E): 1=executable (code), 0=data.
;   Bit 2 (DC): Direction (data: 0=grow up, 1=grow down) / Conforming (code).
;   Bit 1 (RW): Readable (code) / Writable (data). 0=read-only data.
;   Bit 0 (A): Accessed. CPU sets this when segment is used. OS can track.
;
; Flags Nibble (bits 55:52):
;   Bit 3 (G): Granularity. 0=limit in bytes, 1=limit in 4KB pages.
;   Bit 2 (D/B): Default operand size. 0=16-bit, 1=32-bit.
;   Bit 1 (L): 1=64-bit code segment (must be 0 for 32-bit).
;   Bit 0 (AVL): Available for OS use.



7.3 Building the GDT


; A minimal but complete GDT for our OS:
align 8
gdt_start:

; ── Null Descriptor (REQUIRED — selector 0x00) ──────────────────────────────
; The CPU requires the first GDT entry to be all zeros. Loading selector 0
; into a segment register causes a #GP exception.
gdt_null:
    dq 0

; ── Code Segment Descriptor (selector 0x08) ─────────────────────────────────
; Base=0, Limit=4GB, Ring 0, 32-bit, executable, readable
gdt_code:
    dw 0xFFFF        ; Limit [15:0] = 0xFFFF
    dw 0              ; Base [15:0] = 0
    db 0              ; Base [23:16] = 0
    db 10011010b     ; Access: P=1, DPL=00, S=1, E=1, DC=0, RW=1, A=0
    db 11001111b     ; Flags: G=1 (4KB), D/B=1 (32-bit), L=0, AVL=0 | Limit[19:16]=0xF
    db 0              ; Base [31:24] = 0

; ── Data Segment Descriptor (selector 0x10) ─────────────────────────────────
; Base=0, Limit=4GB, Ring 0, 32-bit, writable
gdt_data:
    dw 0xFFFF
    dw 0
    db 0
    db 10010010b     ; Access: P=1, DPL=00, S=1, E=0 (data), DC=0, RW=1, A=0
    db 11001111b     ; Same flags as code segment
    db 0

; ── User Code Segment (selector 0x18, will be used in Chapter 15) ───────────
gdt_user_code:
    dw 0xFFFF
    dw 0
    db 0
    db 11111010b     ; Access: P=1, DPL=11 (Ring 3!), S=1, E=1, RW=1
    db 11001111b
    db 0

; ── User Data Segment (selector 0x20) ───────────────────────────────────────
gdt_user_data:
    dw 0xFFFF
    dw 0
    db 0
    db 11110010b     ; Access: P=1, DPL=11, S=1, E=0, RW=1
    db 11001111b
    db 0

gdt_end:

; GDT Descriptor (used by LGDT instruction)
gdt_descriptor:
    dw gdt_end - gdt_start - 1   ; Limit = size - 1
    dd gdt_start                  ; Base address

; Segment selector constants:
CODE_SEG       equ gdt_code - gdt_start       ; 0x08
DATA_SEG       equ gdt_data - gdt_start       ; 0x10
USER_CODE_SEG  equ gdt_user_code - gdt_start  ; 0x18
USER_DATA_SEG  equ gdt_user_data - gdt_start  ; 0x20



7.4 The Switch to Protected Mode


The transition requires exact sequencing. Any mistake causes an immediate triple fault and CPU reset.



; Prerequisites checklist:
; [1] A20 is enabled (verify!)
; [2] GDT is constructed and GDT descriptor is filled
; [3] Interrupts are disabled (CLI)
; [4] A far jump target is prepared in 32-bit code

switch_to_protected_mode:
    cli                      ; [1] Disable interrupts
    lgdt [gdt_descriptor]    ; [2] Load GDT register

    mov eax, cr0
    or eax, 0x00000001      ; [3] Set PE (Protection Enable) bit
    mov cr0, eax

    ; [4] FAR JUMP — this is MANDATORY
    ; The far jump does THREE critical things:
    ;   a) Flushes the CPU instruction pipeline (old 16-bit decode is discarded)
    ;   b) Loads CS with the code segment selector (0x08)
    ;   c) Sets EIP to the offset within that segment
    jmp CODE_SEG:protected_mode_entry

; ══════════════════════════════════════════════════════════════════════════════
; FROM THIS POINT FORWARD, WE ARE IN 32-BIT PROTECTED MODE
; ══════════════════════════════════════════════════════════════════════════════
[BITS 32]
protected_mode_entry:
    ; Set up all data segment registers
    mov ax, DATA_SEG
    mov ds, ax
    mov es, ax
    mov fs, ax
    mov gs, ax
    mov ss, ax

    ; Set up 32-bit stack at 0x90000
    mov ebp, 0x90000
    mov esp, ebp

    ; Now safe to call 32-bit code!
    call kernel_main

    ; If kernel_main returns, halt forever
.halt:
    cli
    hlt
    jmp .halt

8.1 Linker Scripts


When building a kernel from multiple assembly files, we use a linker script to control the exact memory layout. The linker script specifies where each section (.text, .data, .rodata, .bss) is placed in memory, the entry point address, and symbol definitions that our code can reference.



; ═══════════════════════════════════════════════════════════════════════════════
; kernel.ld — Linker script for our 32-bit kernel
; ═══════════════════════════════════════════════════════════════════════════════
ENTRY(kernel_main)           ; Entry point symbol

SECTIONS
{
    /* Kernel starts at 1 MB physical */
    . = 0x00100000;

    /* .text section: code, read-only */
    .text : {
        *(.text)            /* All .text sections from all object files */
        *(.text.*)
    }

    /* .rodata: read-only data (string literals, constants) */
    .rodata : {
        *(.rodata)
        *(.rodata.*)
    }

    /* .data: initialized read-write data */
    .data : {
        *(.data)
        *(.data.*)
    }

    /* .bss: uninitialized data — must be zeroed at startup */
    .bss : {
        kernel_bss_start = .;
        *(.bss)
        *(.bss.*)
        *(COMMON)           /* Fortran-style common blocks */
        kernel_bss_end = .;
    }

    /* Symbol marking end of kernel */
    kernel_end = .;
}

; ═══════════════════════════════════════════════════════════════════════════════
; kernel_entry.asm — First code executed in the kernel
; ═══════════════════════════════════════════════════════════════════════════════
[BITS 32]
[EXTERN kernel_main]
[EXTERN kernel_bss_start]
[EXTERN kernel_bss_end]

KERNEL_STACK_SIZE  equ 16384     ; 16 KB initial stack

section .text
global kernel_entry

    ; ── Clear BSS section ──────────────────────────────────────────────────
    ; BSS must be zeroed before any code uses uninitialized global variables.
    ; We use REP STOSD for maximum speed: 4 bytes per iteration.
    mov edi, kernel_bss_start
    mov ecx, kernel_bss_end
    sub ecx, kernel_bss_start
    add ecx, 3           ; Round up to nearest 4
    shr ecx, 2           ; ECX = number of dwords
    xor eax, eax         ; EAX = 0 (value to store)
    cld                  ; Forward direction
    rep stosd            ; Store ECX zeros starting at EDI

    ; ── Set up kernel stack ────────────────────────────────────────────────
    mov esp, kernel_stack_top
    mov ebp, esp         ; Initialize frame pointer

    ; ── Call main kernel function ──────────────────────────────────────────
    call kernel_main

    ; If kernel_main returns (it should never), halt.
.halt:
    cli
    hlt
    jmp .halt

section .bss
align 16
kernel_stack:
    resb KERNEL_STACK_SIZE
kernel_stack_top:

A proper kernel needs multiple stacks: one for the main kernel thread, one for interrupts, and one per task. The stack is used for: function call return addresses and local variables, interrupt handler context saving, system call parameter passing, and temporary register spill space.

8.4 BSS Clearing: Why and How


The BSS section contains uninitialized global and static variables. The C standard requires that these be zero-initialized before main() runs. In assembly, we must do this manually. The REP STOSD instruction is the fastest way — it stores 4 bytes per clock cycle on modern CPUs. For a typical kernel BSS of a few KB, this takes microseconds.

9.1 IDT Architecture


In Protected Mode, the old Real Mode Interrupt Vector Table (IVT) is replaced by the Interrupt Descriptor Table (IDT). The IDT can contain up to 256 entries, each 8 bytes (or 16 bytes for 64-bit), describing the handler for each interrupt vector. The CPU references the IDT through the IDTR register, loaded with the LIDT instruction.



; IDT Entry (8 bytes):
;
;  Bits: 63          48 47     40 39 32 31      16 15        0
;       +--------------+--------+------+----------+-----------+
;       │Offset [31:16]│  Flags │ Zero │ Selector │Offset[15:0]│
;       +--------------+--------+------+----------+-----------+
;
;  Selector: Code segment selector (0x08 for our kernel code)
;  Flags byte: P(1) | DPL(2) | 0 | Gate Type(4)
;    P = Present (must be 1)
;    DPL = Descriptor Privilege Level (0 for kernel-only, 3 for user-accessible)
;    Gate Type: 0xE = 32-bit Interrupt Gate, 0xF = 32-bit Trap Gate
;
; Interrupt Gate (0x8E): Clears IF on entry (prevents nested IRQs automatically)
; Trap Gate (0x8F): Does NOT clear IF (used for debugging, allows nested interrupts)

Chapter 10
The 8259A PIC & Hardware IRQ Handling


Remapping hardware interrupts: from IRQ to IDT vector



10.1 PIC Architecture


The Intel 8259A Programmable Interrupt Controller (PIC) manages hardware interrupts. There are two cascaded PIC chips: the Master (IRQ0-IRQ7) and the Slave (IRQ8-IRQ15). The slave is connected to the master's IRQ2 line. When a device signals an interrupt, the PIC translates it into a CPU interrupt vector.



By default, the PIC maps IRQ0 to vector 8, IRQ1 to vector 9, and so on. This conflicts with CPU exception vectors 8-15! We must remap the PIC so that IRQs use vectors 32-47 (0x20-0x2F).



10.2 Remapping the PIC


PIC1_CMD   equ 0x20        ; Master command port
PIC1_DATA  equ 0x21        ; Master data port
PIC2_CMD   equ 0xA0        ; Slave command port
PIC2_DATA  equ 0xA1        ; Slave data port
PIC_EOI    equ 0x20        ; End-of-Interrupt command

ICW1_INIT  equ 0x11        ; Initialize + expect ICW4
ICW4_8086  equ 0x01        ; 8086 mode (not 8080/85)

pic_remap:
    pusha

    ; Save masks
    in al, PIC1_DATA
    push eax
    in al, PIC2_DATA
    push eax

    ; ICW1: Start initialization sequence
    mov al, ICW1_INIT
    out PIC1_CMD, al
    call io_wait
    out PIC2_CMD, al
    call io_wait

    ; ICW2: Vector offsets
    mov al, 0x20         ; Master: vectors 0x20-0x27 (32-39)
    out PIC1_DATA, al
    call io_wait
    mov al, 0x28         ; Slave: vectors 0x28-0x2F (40-47)
    out PIC2_DATA, al
    call io_wait

    ; ICW3: Cascade configuration
    mov al, 0x04         ; Master: slave connected to IRQ2 (bit 2)
    out PIC1_DATA, al
    call io_wait
    mov al, 0x02         ; Slave: cascade identity = 2
    out PIC2_DATA, al
    call io_wait

    ; ICW4: 8086 mode
    mov al, ICW4_8086
    out PIC1_DATA, al
    call io_wait
    out PIC2_DATA, al
    call io_wait

    ; Restore saved masks
    pop eax
    out PIC2_DATA, al
    pop eax
    out PIC1_DATA, al

    popa
    ret

io_wait:
    jmp $+2
    jmp $+2
    jmp $+2
    ret



10.3 End of Interrupt (EOI)


After handling an IRQ, you must send an EOI command to the PIC. Failure to do so will prevent future interrupts from that source.



; Send EOI for IRQ0-IRQ7:
mov al, PIC_EOI
out PIC1_CMD, al

; Send EOI for IRQ8-IRQ15 (must send to BOTH!):
mov al, PIC_EOI
out PIC2_CMD, al       ; Send to slave first
out PIC1_CMD, al       ; Then to master



10.4 PIC Cascading Explained


The master PIC has 8 IRQ lines (0-7). IRQ2 is not connected to a device directly — instead, it connects to the slave PIC. This "cascade" allows the system to support 15 total IRQs (IRQ0-IRQ15, with IRQ2 being reserved for the cascade). When the slave PIC receives an interrupt (IRQ8-IRQ15), it signals the master through IRQ2. The master then generates a single interrupt to the CPU. This is why we must send EOI to both PICs for IRQ8-IRQ15.
